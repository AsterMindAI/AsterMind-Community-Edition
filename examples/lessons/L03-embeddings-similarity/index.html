Why text has to become a vector

Models do math. Math wants numbers, not strings. So before any classifier or store can compare two phrases, the phrases have to become vectors of numbers — same length, every time.

"hello"  → [0.12, 0.04, 0.00, 0.31, ..., 0.07]   (length: 30)
"bonjour" → [0.00, 0.18, 0.22, 0.00, ..., 0.03]   (length: 30)

Same length is critical: that's how you can compare them. The encoder is the function that turns string → fixed-length vector.

Today's encoder counts character occurrences. "hello" has lots of L's; the L slot in the output gets a high number. That's the whole idea. Crude, but it works.

Cosine: how aligned are two arrows?

Once you have two vectors, the simplest measure of "how similar" is the angle between them — how much they point in the same direction.

The cosine of θ is what we use:

  • cos(0°) = 1.00 — same direction (very similar)
  • cos(90°) = 0.00 — perpendicular (unrelated)
  • cos(180°) = -1.0 — opposite (very dissimilar)

For text vectors with non-negative entries (like character counts), the score is bounded between 0 and 1 — never negative.

UniversalEncoder — string → vector

const { UniversalEncoder } = window.astermind;

const encoder = new UniversalEncoder({
  mode: 'char',     // character mode (the simplest)
  charSet: 'abcdefghijklmnopqrstuvwxyz ',
  maxLen: 30,
});

const v = encoder.encode("hello");
// v === [0.0, 0.0, ..., 0.5, 0.0, 0.5, ...]  (length: encoder.getVectorSize())

The output length is fixed (it depends on the config), independent of the input string's length. Long strings get truncated; short strings get padded.

The encoder also has a mode: 'token' option that splits on whitespace instead of characters. We'll stay in character mode today — it's robust to typos and unseen words.

EmbeddingStore — keep them, query them

const { EmbeddingStore } = window.astermind;

const store = new EmbeddingStore(encoder.getVectorSize());

store.add({ id: 'g1', vec: encoder.encode("hello there"),    meta: { lang: 'en' } });
store.add({ id: 'g2', vec: encoder.encode("good morning"),   meta: { lang: 'en' } });
store.add({ id: 'g3', vec: encoder.encode("bonjour mon ami"), meta: { lang: 'fr' } });
// ... 7 more

const queryVec = encoder.encode("hi");
const top3 = store.query(queryVec, 3, { metric: 'cosine' });
// top3 === [
//   { id: 'g1', score: 0.61, ... },
//   { id: 'g2', score: 0.52, ... },
//   { id: 'g4', score: 0.41, ... },
// ]

That's all the API you need today: add, query. The store handles the cosine math; you just give it the vectors.

Two kinds of similarity

What you built today: lexical similarity. Strings that share characters get high scores. Useful for fuzzy matching, typo tolerance, autocomplete.

What you didn't build: semantic similarity. Phrases that mean the same thing get high scores even if they share no characters. "hi" ≈ "salut" ≈ "hola". That requires a different kind of encoder — typically one trained on huge text corpora.

Both have legitimate uses. Don't reach for the heavier semantic version when lexical is enough.

In later lessons you'll see EncoderELM — a small trainable encoder that learns task-specific similarity. It's not a substitute for big models, but it's a real step beyond pure character counts.
