When retraining isn't an option

L02 trained a classifier in one shot: collect data, call train, predict. That works when:

  • You have all the training data up front, and
  • The world isn't changing under you.

Real systems often don't have either. Examples:

  • A user types a message; the autocomplete should adapt to their style as they go.
  • Sensors stream data; new readings arrive every second; you can't pause for batch retraining.
  • A game opponent should learn the player's strategy round by round (Thomas's capstone).

Online learning is the answer: update the model one example (or small batch) at a time, without retraining from scratch.

RLS, plainly

OS-ELM (the algorithm under the hood) uses Recursive Least Squares — RLS — to do online updates.

The intuition without the algebra: the model keeps two pieces of state instead of just the weights.

  • β — the weights (what the model has learned so far).
  • P — a matrix that tracks how confident the model is about each direction in feature space. Roughly: "I've seen this before; small update needed" vs "this is new; pay attention."

When a new example arrives, RLS uses P to figure out how much to nudge β — efficiently, without re-touching all prior examples.

You don't need to read the math today. You just need to know that updates are O(1) in the dataset size — they don't get slower as the model sees more data.

The forgetting factor λ

RLS gives equal weight to every past sample by default. That's fine when the world is stable. It's bad when the world changes — old data starts to lie about what the world looks like now.

The forgetting factor λ ∈ (0, 1] exponentially down-weights old samples:

  • λ = 1.0 → never forget. Every sample weighs the same forever.
  • λ = 0.99 → mild forgetting. After 100 samples, the oldest carries ~37% the weight of the newest.
  • λ = 0.9 → aggressive forgetting. After 10 samples, the oldest carries ~35% weight.

It's a tradeoff:

  • Lower λ = adapts faster to change, but accuracy is jumpier when the world is stable (over-fits to recent noise).
  • Higher λ = stable on stationary data, slow to recover from drift.

You'll feel this tradeoff hands-on in You-Try 3.

init → update → predict

const { OnlineELM } = window.astermind;

const ol = new OnlineELM({
  inputDim: 2,
  outputDim: 2,
  hiddenUnits: 16,
  forgettingFactor: 0.99,   // lambda
});

// Step 1: seed batch — RLS needs initial P estimate.
ol.init(seedX, seedY);

// Step 2: stream new examples one at a time.
ol.update([[0.7, 0.4]], [[0, 1]]);  // single sample as 1×D / 1×K matrices

// Step 3: predict any time.
const probs = ol.predictProbaFromVectors([[0.5, 0.5]]);
// probs === [[0.12, 0.88]]  // softmax-normalized class probabilities

Three rules to remember:

  1. init requires a small batch (we'll use 4 points) before any update can be called.
  2. update takes 2D arrays even for single samples — wrap your row in another array.
  3. predict returns probability rows; use argmax to get a class label.

This is Thomas's capstone, in pieces

The 2D streaming demo you just built is structurally identical to RPS-with-adaptive-opponent:

  • Each round = one streamed example.
  • Player's history (last few moves) = the input vector.
  • Player's actual move that round = the label to update against.
  • Concept drift = the player suddenly switching strategies (rock-spammer becomes scissors-spammer).
  • Forgetting factor = how quickly the agent should chase strategy shifts.

For an RPS opponent that feels alive but not chaotic, λ ≈ 0.95 is a sensible starting point. Tune from there.
