Every prediction has a confidence

L02 had us calling predict and reading the top label. But the API returns more than that:

const result = clf.predict("login slow today", 2);
// result === [
//   { label: "urgent", prob: 0.62 },   // ← this is the confidence
//   { label: "low",    prob: 0.38 },
// ]

The prob field is the model's confidence. It's been there all along; we just ignored it. Today we use it.

For a binary classifier (two classes), confidence is in [0.5, 1.0]:

  • Near 1.0 — model is sure about the top class.
  • Near 0.5 — model is on the fence; could go either way.

Confidence ≠ correctness. A confident prediction can still be wrong; a borderline one can be right. But on average, confidence correlates with reliability — and that's enough to be useful.

Two ways to be wrong

Imagine a support-ticket triage system. The model flags tickets as "urgent." Two distinct mistakes:

  • False positive (FP): flagged "urgent", actually low priority. → wastes the on-call's time.
  • False negative (FN): flagged "low", actually urgent. → real problem missed.

Two metrics that pin down which kind of error you're making:

  • Precision = of the items I flagged urgent, what fraction were actually urgent? (Penalises FP.)
  • Recall = of the truly urgent items, what fraction did I flag? (Penalises FN.)

You can't max both. Improving one usually hurts the other. The threshold knob is how you choose.

The threshold knob

A threshold τ in [0.5, 1.0] says: "only accept the urgent prediction if its confidence is at least τ." Otherwise, treat as low priority (or "I don't know" — depends on your domain).

function decide(text, tau) {
  const result = clf.predict(text, 2);
  const top = result[0];
  if (top.label === 'urgent' && top.prob >= tau) return 'urgent';
  return 'low';
}

Two extreme cases:

  • τ = 0.50 → accept any "urgent" the model leans toward. Highest recall, lowest precision. The on-call gets every alert.
  • τ = 0.99 → only the most confident "urgent" predictions count. Highest precision, lowest recall. Quiet alerts; some urgents slip through.

Production systems pick somewhere in between. The "right" τ depends on the cost of FP vs FN in your domain.

What the tradeoff looks like

Sweep τ from 0.5 to 1.0 and plot (recall, precision) at each point. You get the precision-recall curve:

τ = 0.50  →  precision 0.70  recall 1.00
τ = 0.70  →  precision 0.83  recall 0.83
τ = 0.85  →  precision 0.92  recall 0.67
τ = 0.95  →  precision 1.00  recall 0.33

The shape isn't a straight line — it depends on the dataset and the model. A flat-ish curve in the upper right is what you want; a steep dropoff means choosing τ is fragile.
